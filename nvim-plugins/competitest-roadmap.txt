## ✅ PR #1: Submit Provider System (COMPLETED)

**Branch:** feature/submit
**Status:** Merged to fork, PR #82 submitted to upstream (https://github.com/xeluxee/competitest.nvim/pull/82)

### What Was Built

**Generic Submit Provider System** - Extensible architecture for competitive programming judges

**Features:**
- Provider interface with async callbacks (submit, submit_multiple, get_status)
- Kattis provider implementation:
  - `.kattisrc` parsing (supports both `:` and `=` delimiters)
  - Language auto-detection from file extension with fallback
  - Problem ID from filename (lowercase)
  - Mainclass detection for Java/Kotlin/Scala
  - Multi-file submissions via file picker (snacks.nvim/nui.nvim)
  - Live status polling with test case progress
- Commands: `:CompetiTest submit [provider]`, `:CompetiTest submit_multiple [provider]`
- Notification flow: "Submitting..." → "Submitted! - View: URL" → "Accepted" / "Wrong Answer (failed X/Y)"
- Async/non-blocking: All HTTP via vim.loop.spawn(), no vim.fn.system()
- Production-ready: QA tested, proper resource cleanup (timers, pipes, handles)

**Files Added:**
- `lua/competitest/submit/init.lua` - Provider registry and orchestration
- `lua/competitest/submit/ui.lua` - File picker UI
- `lua/competitest/submit/providers/kattis.lua` - Kattis implementation
- Updates to: `commands.lua`, `config.lua`, `utils.lua`

**Repository:** https://github.com/yuuhikaze/competitest.nvim (fork)

## 🔮 Future Enhancements (Post-PR #3)

### 1. Auto-detect Submit Provider from Metadata

**Current:** Explicit provider selection: `:CompetiTest submit kattis`

**Future:** Automatic detection from competitive-companion metadata

**How it works:**
1. When receiving problems via competitive-companion, persist metadata alongside testcases:
   - `url`: "https://open.kattis.com/problems/hello"
   - `group`: "Kattis - Practice"
   - `problem_id`: extracted from URL

2. Store in testcases file or .competitest.lua:
   ```lua
   {
     metadata = {
       url = "https://open.kattis.com/problems/hello",
       judge = "Kattis",
       problem_id = "hello",
     },
     testcases = { ... }
   }
   ```

3. When user runs `:CompetiTest submit` (no provider specified):
   - Load stored metadata
   - Each provider implements `detect(metadata) -> boolean`
     - Kattis: check if url contains "kattis.com"
     - Codeforces: check if url contains "codeforces.com"
   - First matching provider auto-selected
   - Fallback: prompt user if no match

**Benefits:**
- Seamless workflow: receive → code → test → submit (no manual provider selection)
- Problem ID automatically available
- Works great with competitive-companion integration

**Implementation:**
- Extend testcases storage format to include optional metadata field
- Update receive.lua to persist CCTask metadata
- Add `Provider.detect()` to interface in submit/init.lua
- Backwards compatible: if no metadata, fall back to explicit selection

## Testing Your Fork

Replace competitest.nvim config with local fork:

```lua
-- For lazy.nvim
{
  dir = '/home/user/plugin/kattis.nvim/misc/competitest.nvim',
  name = 'competitest.nvim',
  dependencies = 'MunifTanjim/nui.nvim',
  config = function()
    require('competitest').setup({
      submit = {
        kattis = {
          config_file = vim.fn.expand('~/.kattisrc'),  -- optional, auto-detected
        },
      },
    })
  end
}
```

Then test:
- `:CompetiTest submit kattis` - single file
- `:CompetiTest submit_multiple kattis` - multi-file picker
